# Split the API by audience: `/admin` and `/app`

Inside the version prefix (`/api/v1`), routes are grouped under two audience routers in `src/api/routers.py`: `/admin` for back-office endpoints governed by the Admin model and RBAC, and `/app` for mobile/frontend endpoints aimed at end Users. Each domain router (`auth`, `admins`, `rbac`, `logs`, `settings`, …) is mounted under exactly one audience. `/app` is reserved but intentionally empty until the first app-facing module — and its own User-side auth — exists.

We chose this because the two audiences differ in nearly every cross-cutting concern: who authenticates (Admin + RBAC vs User + phone/social), what authorization applies, and how the surface is allowed to evolve. Keeping them as sibling router trees makes those differences structural rather than per-route conventions, and lets each side grow its own middleware, auth dependency, and docs grouping. The trade-off is that the audience of an endpoint is fixed by where it is mounted: moving a resource from admin-only to app-facing (or exposing it to both) means a path change, which is hard to reverse once clients integrate against it. A shared resource needed by both audiences must be deliberately mounted twice rather than appearing at a single neutral path.
