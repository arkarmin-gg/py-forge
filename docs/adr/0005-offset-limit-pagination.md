# Paginate with offset/limit, not cursors

List endpoints return a `Page[T]` envelope (`items`, `total`, `limit`, `offset`) built by `src/pagination.py`. Callers inject `pagination_params` (validated `limit`/`offset` query parameters, bounded by `MAX_PAGE_SIZE`) and hand `paginate()` both a data statement and a matching count statement; the helper runs the windowed query plus a `COUNT` and assembles the page. Filters are applied to both statements (see `src/query_filters.py`) so `total` reflects the filtered set.

We chose offset/limit because the consumers are admin-console list views that want jump-to-page navigation and a visible total count — both of which fall out naturally from `OFFSET`/`LIMIT` + `COUNT`, and neither of which cursor pagination gives cheaply. It is also the least surprising shape for a starter: any client can request page N without threading an opaque cursor. The trade-off is the well-known offset cost — deep pages scan and discard `OFFSET` rows, and the count query grows expensive on large tables — and the usual consistency caveat that rows inserted or deleted between page fetches can shift the window. If a high-volume, deeply-paged endpoint later needs it, switch that specific endpoint to keyset/cursor pagination rather than changing the default for everything.
